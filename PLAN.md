Forward-looking roadmap toward a **polished open-source release**. For how the repo works *today*, see [AGENTS.md](AGENTS.md); for user-facing usage see [README.md](README.md).

> **Living document — update it often.** As phases land, check items off and revise. When a decision changes, edit here in the same change. Created 2026-07-14.
>
> **Provenance:** this plan was synthesized from a multi-model planning round run *through ClaudeCollab itself* — the same brief sent to `openai/gpt-5.6-sol`, `opencode/deepseek-v4-flash-free`, and `opencode/nemotron-3-ultra-free` via `collab/ask.sh`, then reconciled by Claude. Dogfooding validated that the loop works end-to-end (Phase 0 partially complete).

## Vision (settled)

A **"bring other minds to bear"** tool where model *diversity is the point*, not decoration. Four reasons to reach for another model: (a) more angles from differently-trained models, (b) cost / offloading grunt work, (c) verification against groupthink, (d) capability Claude lacks.

**Principles:**
1. **Claude's role is contextual** — coordinator/"boss" by default; *peer* when explicitly invoked via `/collab:collaborate`.
2. **Anti-goal: Claude must not dismiss the other models.** Engagement must be *observable* — see the collaboration contract below. Diversity Claude ignores is worse than useless.
3. **Anthropic models — eligibility depends on mode.** When Claude *authors* the work (collaborator/direct mode) it already supplies the Anthropic view, so external opinions lean **non-Anthropic** for real diversity. When Claude only *coordinates* (delegates the work out and synthesizes, without authoring it itself), its reasoning isn't the deliverable, so an **Anthropic model is eligible** among the fanned-out agents — otherwise that perspective is absent from the actual work. Never denied by default; this is selection preference, not permission. No API keys stored or managed by this repo.

### Settled design decisions (2026-07-14)
- **`/collab:collaborate` is an explicit command**, not inferred from phrasing (inference is ambiguous and untestable; the explicit command *is* the "work with others" signal).
- **`/consensus` → `/collab:panel`** — "consensus" manufactures false agreement; the value is preserved disagreement.
- **`/collab:research` is gated** — build it only after proving opencode + model actually perform live web search; until then, document `/collab:consult "research X"` as the workaround.
- **License = MIT.**
- **Model policy (allow / ask / deny)** — a *configurable per-developer mechanism* (see below), not a fixed list. Ships **default-allow** with commented examples; each developer gates whatever specific models they want.

### `/collab:collaborate` memory = Option B, session continuation (2026-07-15)
- **Decision:** `/collab:collaborate` uses opencode **session continuation** (`-s <session-id>`, id captured from the first call's `--format json`), **not** Claude re-packaging the transcript each turn.
- **Why:** the peer's prior turns live in opencode's session, so **Claude never re-transmits the other model's words** — it only sends its own new turn. Fidelity becomes a **construction guarantee**, not a matter of Claude's discipline. Option A was rejected because its "pass the peer's words verbatim" rule relies on Claude actually doing so, and Claude is known to quietly edit/paraphrase outputs — an unenforceable, trust-based guarantee. (A wrapper-owned mechanical transcript was also considered; B was preferred as the cleaner construction guarantee.)
- **Logging:** a full back-and-forth record is still kept by capturing each peer response (`ask.sh` stdout) plus Claude's sent turns — a parallel log, slightly harder to assemble since opencode's session is the authoritative store.
- **Requires** the session plumbing (capture session id via `--format json`, thread `-s <id>`, `--fork` to branch, hard turn cap) — promoted from "Phase 4 / later" into the `/collab:collaborate` build itself.
- **Provenance:** settled via a live `/collab:collaborate`-contract dogfood — independent view (leaned A) → peer `gpt-5.5` (also A, but surfaced the "Claude-as-curator" risk) → user flagged verbatim-by-Claude is unenforceable → resolved to B (verbatim-by-construction).

### Guiding priority (2026-07-15)
- **Done well over done fast.** There is no pressure to ship a quick v0.1. Correctness, reliability, and a good experience win over scope-cutting for speed. Infrastructure that makes it work well — **background servers/daemons, a persistent warm process, lifecycle management — is explicitly in scope.** (This overrides the earlier "no background daemons" scope cut, which was a planning-round assumption, not a requirement.)

### Transport note (2026-07-15) — the "latency crisis" was a stdin bug, not opencode
- **What actually happened:** `opencode run` **blocks waiting on stdin** when stdin is a non-TTY pipe — exactly what Claude Code's Bash tool provides. Every Claude-invoked call hung ~60s until killed. This was misdiagnosed at length as a degraded backend, a per-call instance-boot cost, TUI/session contention, and Bash-sandbox interference — **all wrong.** Interactive terminals are a TTY and never hit it, which is why it only bit the wrapper when Claude drove it (the user's TUI was always fast — the data point that should have been believed sooner).
- **Fix (shipped in `ask.sh`):** redirect the opencode call's stdin from `/dev/null` (immediate EOF), plus a `$COLLAB_TIMEOUT` backstop (default 300s) that fails loudly instead of hanging. With the fix, opencode responds in **~3–5s**. There is **no inherent opencode latency problem and no backend degradation.**
- **Consequence:** a persistent `opencode serve`/daemon is **NOT needed for latency.** A warm server might still be worth it *later*, purely for multi-turn `/collab:collaborate` continuity/economy — but that's an optional enhancement, evidence-gated, not a fix. Daemons remain permissible (see Guiding priority) but are unneeded now.
- **Lesson:** when one clean data point contradicts the theory, test the invocation path early — don't explain the data point away.

## The collaboration contract ("engage, don't dismiss")

Bake into every command prompt so engagement is observable, not promised:
1. **Independent first pass** — Claude forms a preliminary view *before* calling the other model, and does not include it in the external prompt (anti-anchoring; gives a baseline to notice real updates).
2. **Structured external contribution** — ask the other model for: direct recommendation, 2–4 key supporting claims, evidence, the strongest objection to its own view, and "what a competent Claude analysis will most likely overlook."
3. **Disposition each material point** — Claude marks Adopt / Adapt / Reject / Defer with a reason. "Reject" requires evidence, a concrete tradeoff, or an identified error — never "I prefer another approach." Then state: what changed in my view, what I'm not adopting and why, what's unresolved.
4. **One targeted rebuttal** — for disagreement that materially changes the decision, one follow-up call. No unbounded debate loops.
5. **Attribution** — always print the exact provider/model id; delimit the external response; in `/collab:panel` never collapse disagreement into "the models agree" unless they do.

## Model policy (allow / ask / deny)

Not every reachable model should be usable by these commands without a check. A declarative policy governs which models `/collab:consult`, `/collab:panel`, `/collab:review`, `/collab:research`, `/collab:delegate`, `/collab:collaborate` may use, in three tiers:

- **allow** — use freely.
- **ask** — Claude must confirm with the user *before* using this model.
- **deny** — never use; the wrapper hard-refuses.

**Defense in depth (both layers):**
- *Claude-side (command prompts):* before choosing a model, Claude consults the policy — never proposes a `deny` model, and stops to ask the user when the intended model is `ask`-tier. Always names the exact model id chosen.
- *Wrapper-side (`ask.sh`):* independently checks the requested `-m` against the policy and **refuses to run a `deny`-tier model** (a hard backstop against mistakes or prompt injection, since a prompt can't talk `ask.sh` out of it). Option: `ask`-tier requires a `COLLAB_CONFIRMED=1` flag that Claude sets only after actually asking.

**Format:** a simple, greppable, glob-pattern file (`collab/models.policy`), **first match wins**, **default-allow**. It ships permissive with commented examples; each developer uncomments/edits to gate the specific models *they* care about. Shipped default:

    # collab/models.policy — governs which models these commands may use.
    # First match wins; patterns are globs on the full provider/model id.
    # Uncomment / edit to gate specific models. Tiers: deny | ask | allow.
    #
    #   deny   *-sol*             # example: never use Sol variants
    #   ask    *fable*            # example: confirm before using Fable
    #   ask    openai/gpt-5.6-*   # example: confirm before a specific family
    allow  *                      # default: everything is allowed

Because a `deny` rule can sit above `ask`/`allow`, a broad "ask before family X" rule can still hard-deny a specific variant. The point is the *mechanism*: any developer expresses their own allow/ask/deny preferences without touching code. **Per-user config is now file-based & non-committed (DONE 2026-07-15).** Env vars were rejected as the *home* for config — a Claude-driven setup command runs in a subshell and can't durably `export` into your shell (only useful as one-off overrides). So two git-ignored files hold it, both effective immediately:
- **Policy** — `ask.sh`/`doctor.sh` resolve `$COLLAB_POLICY` → `collab/models.policy.local` → committed `collab/models.policy`.
- **Preferred models** — `collab/collab.conf.local` (`KEY=value`, template `collab.conf.example`, parsed not sourced): `ask.sh` reads `COLLAB_MODEL`, `panel-models.sh` reads `COLLAB_MODELS`, precedence flag/args → env override → file → default.
**`/collab:configure`** (`.claude/commands/collab/configure.md`) is the guided setup: interviews (deny/ask, preferred default + panel set, personal-vs-shared scope), writes both files, validates via `doctor.sh` + `--dry-run`. Unit-tested: a ruleful `.local` policy is preferred over default and `$COLLAB_POLICY` overrides both; an empty/comment-only local file falls through to the committed policy; conf-file model is used when env is unset and `$COLLAB_MODEL`/`$COLLAB_MODELS` override the file.
- **Hardened by a 2nd multi-model review (2026-07-15, dogfooded via `/collab:consult` on openai/gpt-5.5 + opencode/nemotron, verified against code).** It caught **two policy fail-opens** — an empty/comment-only `models.policy.local` silently voided a committed deny set, and an unreadable policy file failed open — both fixed: `.local` is preferred only if it has ≥1 rule (else fall through to committed, so an empty personal file can't void a shared deny), and `policy_tier` fails **closed** (deny + loud error) when the resolved policy exists but is unreadable. Also fixed: `doctor.sh` now policy-checks each `/collab:panel` member (a denied member was passing silently) and labels the config source correctly under `$COLLAB_CONF`; the conf parser strips inline `# comments` and refuses a leading-`-` model id (opencode-flag-injection guard); a test asserts the triplicated `conf_get` stays byte-identical (drift guard); `/collab:configure` allowed-tools gained `panel-models.sh`. (A reviewer's "CRITICAL key-collision" was verified **false** — `COLLAB MODEL=` doesn't match `COLLAB_MODEL`.) Suite 40 → 45.

### Global (user-level) install — DESIGNED + BUILT 2026-07-17

**The ask (maintainer, 2026-07-17):** the current installer drops the payload into a single project (`install.sh` → a `--dest`). Can it instead live in the user's **global** Claude config (`~/.claude`) and work from *any* project, the way user-level slash commands, agents, and settings already do? Yes — and less invasively than it first looks, because the couplings were traced live rather than guessed.

**What already makes this feasible (don't re-solve it).** The scripts resolve their **own** siblings via `dirname "$0"`, not the CWD: policy/config/log-dir/agent-def-checks in `ask.sh` (`ask.sh:120-123,172,258,296`), `log.sh` (`here="$(cd "$(dirname "$0")" && pwd)"`, `LOG_DIR="$here/logs"` — `log.sh:107,140`), and `panel-models.sh:30`. So a single fixed copy of `collab/` at an absolute location already finds its own policy, config, and logs from anywhere. And `/collab:delegate`'s git snapshot walks the CWD (`find . -xdev`, `ask.sh:587`) — already correctly the *project you're in*, not the install dir, so it needs no change.

**Target layout.**
- Scripts → `~/.claude/collab/` (one canonical copy; sibling resolution already works).
- Agents → the opencode **global** config dir (`~/.config/opencode/agent/collab-*.md`). This is not cosmetic — it is the actual runtime resolution path (see change A).
- Commands → `~/.claude/commands/collab/*.md` (Claude Code user-level commands, namespaced `/collab:<name>` exactly as the per-project install already relies on — see "Command namespacing" below).

**The real changes.**

- **A. opencode must find the agents globally (hard requirement).** `ask.sh` runs `opencode run --agent collab-read …` without `cd`-ing (`ask.sh:434` builds the args; `run_opencode` invokes at `750`/`766`/`807`/`817`), so opencode resolves `--agent` from *its own* config — the CWD project's `.opencode/agent/` or the global opencode config dir. An arbitrary project has neither of ours, so the four defs **must** install to opencode's global agent dir. **VERIFY FIRST** (repo rule: test, don't assume): that the global path is `~/.config/opencode/agent/` (honor `$XDG_CONFIG_HOME`) and that `--agent collab-read` resolves a globally-installed def. AGENTS.md already records that opencode namespaces/reads global agents, but that was for the subdir question, not a proof of global resolution for `run --agent`.

- **B. `ask.sh`'s agent-def existence checks assume the sibling layout — and only govern *fallback*.** Four checks (`ask.sh:258,381,394,408`; the neighbouring `296`/`298` are the watch log-dir default and scope, not def checks) probe `$(dirname "$0")/../.opencode/agent/collab-*.md`. Globally the defs live in the opencode config dir, not a sibling of `collab/`, so they'd wrongly report "missing" and degrade (or, for `--watch`, exit 5). But fixing only these is **not sufficient**: the wrapper does not `cd` and does not resolve `--agent` — opencode does, independently, from its own config. So a `COLLAB_AGENT_DIR` (which must be a `collab.conf.local` value, **not** an env prefix — an env prefix would break the prefix-matched grants) can make the wrapper's existence check pass while opencode resolves a *different* agent; for `collab-watch`, whose check is a no-fallback security gate, that is a silent oversight failure. B is therefore **inseparable from A and from the coexistence ruling below**: the wrapper's belief about which def is enforced must be proven to equal opencode's actual resolution. (Reviewer finding, verified against the code.)

- **C. Command files hardcode `bash collab/ask.sh` — in both the body and the `allowed-tools` grant**, and the grant surface is bigger than one string. Each command carries **6–8 grant variants** — env-prefixed (`COLLAB_COMMAND=… bash collab/ask.sh:*`), `COLLAB_CONFIRMED=1` forms, `COLLAB_RUN_ID=* …` forms, and a command-substitution grant `Bash(RUN=$(bash collab/log.sh new-run:*))` — across `ask.sh`, `log.sh` **and** `panel-models.sh` (consult/panel/workshop/collaborate/witness/configure all grant `log.sh`, not just three). Claude Code matches these as literal prefixes, so every variant must be transformed. Two options:
  - **PATH launcher:** install launchers `collab-ask`/`collab-log`/`collab-panel` so command bodies call a bare name and grants become `Bash(collab-ask:*)` — command files stay static payload. **But this rests on two unproven/fragile things** (reviewer finding): (i) the launcher dir must be on the PATH **Claude Code's Bash tool actually sees** — not guaranteed, and a VERIFY-FIRST gate, not an assumption; (ii) the scripts use bare `dirname "$0"` with **no `$0` canonicalization**, so a *symlinked* launcher makes every sibling (`models.policy`, `collab.conf.local`, `logs/`, the def checks) resolve against the **bin dir**, silently breaking policy/config/log/fallback — a co-located copy in the bin dir breaks it the same way. Launchers would have to be thin wrappers that `exec` the real script by absolute path, or the scripts must `readlink`-canonicalize `$0`.
  - **Templated absolute path:** installer rewrites every grant + invocation to `~/.claude/collab/ask.sh`. Makes command files user-specific, but sidesteps both PATH hazards above — so given (i)/(ii) this is arguably the **safer default**, not the fallback.
  - Both depend on **verifying** Claude Code loads `allowed-tools` frontmatter identically for user-level commands.

- **D. `install.sh --global` mode.** New branch that resolves `~/.claude` and the opencode config dir **following symlinks** (reuse the `pwd -P` resolve-and-follow already used for `--dest`, `install.sh:126-127`); installs scripts/agents/commands to their three roots; installs the PATH launchers; **skips the `.gitignore` block** (home isn't a repo). Honor `$CLAUDE_CONFIG_DIR` (Claude Code's config-dir override) and `$XDG_CONFIG_HOME`. The ownership model (`.install-manifest`/`.install-hashes`) currently assumes a single `$dest` root — global spans three, so it needs absolute-path or per-root records; uninstall likewise across three roots.

### Egress on `collab-read` — SETTLED, first pass implemented (2026-07-15)

**Provenance:** Claude's own (`75e7d87`). A review found `collab-read` could read `.env` **and** reach the network — a real exfiltration channel. The **secret-read denies** answered that finding. The **egress ban** was Claude adding belt to braces, and was never asked for.

The banned combination is one we perform ourselves without comment. And the guarantee is narrower than the docs imply — **`/collab:research` ships read + egress by the maintainer's own decision**, so "this repo cannot leak" was never true; only "this one command cannot". ### Per-call tool approval (bubbling opencode's permissions up to the user) — INVESTIGATED 2026-07-15, POSSIBLE, not built and not queued

**The maintainer's principle:** if Claude Code is auto-approving, opencode should too; if Claude Code is prompting the user, opencode should ask as well. The wrapper must not grant opencode more than the user granted Claude. Sound, and worth building toward. Here is what's true today, all verified live rather than reasoned:

- **`--auto` is a NO-OP for our hardened agents.** It auto-approves permissions in an *ask* state, and our default-deny allowlists leave nothing in that state — every tool resolves to `allow` or `deny` by construction. **Proven:** `collab-build` edits files fine *without* `--auto`. So the earlier claim that "the one real asymmetry with an in-house subagent is `--auto`" was **wrong**: `--auto` isn't granting anything. The agent def is the grant, and it's static, readable, and ours. `--auto` matters only on the `plan`/`build` fallback paths, whose built-ins do have `ask` rules (e.g. `read: *.env → ask`).
- **An `ask` tier does NOT hang on a non-TTY — opencode AUTO-REJECTS it** (`permission requested: bash (echo hi); auto-rejecting`). Fail-closed, no stdin hang. So "make opencode ask" through today's one-shot transport actually means "make opencode deny", which would simply break `/collab:delegate` rather than prompt anybody.
- **opencode has a complete permission API**, so bubbling approvals up is genuinely possible: `GET /permission` (pending requests), `POST /permission/{requestID}/reply` (approve/deny), `GET /event` (SSE), plus `opencode serve`/`--attach`/`--port` and an ACP server. Claude Code has the other half (background bash, `Monitor`, `AskUserQuestion`). Nothing is missing except the work.
- **Claude Code does NOT expose its live permission mode to a subprocess.** `CLAUDECODE`, `CLAUDE_CODE_SESSION_ID`, `CLAUDE_CODE_ENTRYPOINT` and friends are present; nothing about permission mode. `~/.claude/settings.json` has `permissions.defaultMode`, which is readable — but it is the **configured default, not the live mode**: a mid-session shift-tab toggle is invisible, and project/enterprise settings override it. Mirroring off it would be approximate and would couple us to Claude Code's settings schema.
- **Claude must not self-declare the mode.** The obvious shortcut — Claude tells `ask.sh` which mode it's in — is Claude granting itself permission, on self-report, in a project whose entire thesis is that Claude's self-report is the thing under audit. Reject it if it comes up again.

**What building it would take** (the shape, if we do it): flip `collab-build`'s mutating set (`bash`/`edit`/`write`/`patch`) from `allow` to **`ask`** — the `"*": deny` floor stays and keeps everything else denied by construction. Then `--auto` becomes the real switch: passed → today's behaviour; withheld → each tool call needs a human. That requires the bridge (run against `opencode serve`, watch `/event`, surface via `AskUserQuestion`, `POST .../reply`), which turns `ask.sh` from a one-shot into an orchestrator.

**The prompt-fatigue objection is answered — ask once, then remember.** The maintainer's proposal (2026-07-15): *ask, and persist that decision for the session.* opencode supports this natively; verified against the running server's OpenAPI spec:
- **`POST /permission/{requestID}/reply` takes `reply: ['once' | 'always' | 'reject']`** — `always` is the persist.
- **`PermissionRequest` = `{id, sessionID, permission, patterns, metadata, always, tool}`** — decisions can be **pattern-scoped** (`bash(npm test:*)`), like Claude Code's, not just "all bash forever". Follow opencode's own `patterns`/`always` suggestion for granularity: get that wrong and the feature is either useless or fatiguing, and the cache *is* the UX.
- **`PermissionSavedInfo` = `{id, projectID, action, resource}` — `always` persists per PROJECT, not per session**, and outlives the run (`DELETE /api/permission/saved/{id}` to clean up). **So don't use `always`.** Keep a decision cache in `ask.sh` keyed by `$COLLAB_RUN_ID` and always reply `once`: the first `npm test` asks, the rest are answered from the cache. Session-scoped **by construction**, nothing to clean up, opencode's project-wide store untouched.

**The architecture is feasible and the shape is known.** `ask.sh` can't call `AskUserQuestion` — it's a subprocess — but it doesn't need to: **the opencode session stays alive server-side while a permission is pending**, so the loop lives in Claude's turn, not in the script. Start the run in background → poll `GET /permission` → surface via `AskUserQuestion` → `POST …/reply` → the run continues → completion notification. `collab-build`'s mutating set flips `allow` → `ask` (the `"*": deny` floor is untouched and still holds everything else by construction), which makes `--auto` the real switch. Opt-in via `COLLAB_APPROVE=1` in `collab.conf.local` — **user-set, never Claude-set** (see the self-declaration ban above) — so default behaviour is unchanged.

**Status: NOT BUILT. Deferred deliberately (2026-07-15) — noted as a possible thing to work on, not queued.** No decision has been made to build it; the investigation is banked so it doesn't need redoing.

**Honest bounds if it is ever built** — say these out loud rather than overselling:
- It gates only the **tool-native** route. `bash` approved once means the model can do anything *through* that bash call. It is you seeing what is about to run — **not** a containment boundary. The thing it would catch (`curl evil.com`) is better answered by a network firewall.
- It is worth most on the case AGENTS.md already flags: **delegating on a repo you don't trust**. On a trusted repo it is mostly ceremony, and this project has already removed one piece of ceremony (the clean-worktree guard) that was justified with the same instincts.

### Anthropic voice: the subagent transport — BUILT 2026-07-21 (issue #5)

**The gap.** Every collab command reaches models through one transport: `ask.sh → opencode run`. opencode serves only what its own auth exposes, and commonly that is **not** an Anthropic model — so "put Anthropic on the panel" (a role-rule default, or an explicit request under issue #3) was silently **unfulfillable**, and Anthropic got dropped for an availability reason dressed up as a choice. But Claude Code *can* run Anthropic natively — via **subagents** (the Task/Agent tool) — a second transport the collab flows never reached for.

**What was built.** The collab commands now **spawn a Claude subagent as a voice** when an Anthropic perspective is wanted, alongside the opencode `-m` calls, and record its turn in the evidence layer. Concretely: `collab/log.sh subagent-voice` writes a `subagent-voice` entry (`claim:true, captured:false, transport:"claude-subagent"`, text in `claimed_response`), folded into `verify`'s hash-chain contract; `collab-watch.md`/`witness.md` teach `/collab:witness` to audit it as testimony; and `/collab:panel`, `/collab:workshop`, `/collab:consult`, `/collab:collaborate` carry the guidance + the `Task`/`Write`/`log.sh subagent-voice` grants.

**Decision.**
1. **The driver Claude is still the default Anthropic voice.** On a panel Claude runs, Claude already *is* the Anthropic perspective (it writes the synthesis); in *authoring* mode stating that view as a labelled contribution is free and needs no subagent. A separate subagent is for when you want a genuinely **independent** second Anthropic instance (un-anchored on the driver), or when the user asks for one — that's the case the new transport unlocks.
2. **A subagent is just another agent — spawn it when wanted, don't cage it.** Using a Claude subagent as a collab voice is ordinary Claude Code usage; it gets no special hardened profile, tool cage, or secret-scoping gate that a subagent doing any other task wouldn't get. Inventing one is the vendor-asymmetry bias pointed at the Anthropic path (see below). The only honest guidance that remains is *don't pad*: two Claudes from the same weights aren't two independent votes, so spawn for a real independent perspective, not to make a panel *feel* cross-vendor.
3. **A subagent turn is logged as a CLAIM, and that is the one real, honest asymmetry.** There is no non-Claude capture point in the subagent transport (it runs inside Claude's process), so its reply is **transcribed by Claude**, not wrapper-captured like opencode's `raw_response`. So it's recorded in `claimed_response` and flagged `captured:false` — visible to `/collab:witness`, which audits whether `claude-final` represents it consistently rather than confirming it verbatim. This is disclosure, not restriction: the voice is in the log, labelled for what it is.
4. **A "hardened zero-tool profile" was investigated and is BOTH unwanted and unbuildable.** Unwanted: it was Claude's own invention, not the ask — the maintainer's actual request was *"I just want you to also spawn Claude subagents when we want"* (2026-07-21). Unbuildable anyway, verified against current Claude Code docs: (a) a zero-tool subagent is **refused at launch** (v2.1.208+ errors when nothing in `tools` resolves); (b) there is **no innocuous single tool** that launches while reading nothing and reaching nowhere (`TodoWrite` is disabled by default; every other launchable tool reads, mutates, or has egress); (c) Claude Code has **no per-subagent path scoping** — `tools:` is tool-*name* granularity, and `Read(.env)` denies are session-wide, so opencode's per-agent secret map cannot be replicated on a subagent. So the earlier "zero-tool, strictly more contained than any opencode allowlist" target was a fiction. The right answer was never a cage.
5. **Command scope.** All four collab commands are wired. `/collab:consult` and `/collab:collaborate` still *default* non-Anthropic (a second opinion / peer wants a different mind), but can spawn a Claude subagent on request; `/collab:panel` and `/collab:workshop` spawn one whenever the Anthropic angle belongs and is otherwise absent.

**Provenance + the course-correction (preserved, per CLAUDE.md).** The design was first pressure-tested in a `/collab:panel`-style collab (run `20260720T114413Z-22fcc473`: a `Plan` subagent + `openai/gpt-5.5-fast`) and a `/collab:review` (`gpt-5.6-sol` + `deepseek`). That process produced a **wrong** first pass — Claude built an elaborate cage (a "don't spawn / hardened zero-tool profile / ask the user" regime) around a capability the maintainer had simply asked for. The maintainer corrected it directly: they hadn't asked for a locked-down subagent, only the ability to spawn Claude subagents as voices. The cage was removed. The old design run is still a useful artifact — it *demonstrates* the one real asymmetry live: that run's `Plan` subagent reply was never logged, so it wasn't witnessable; the `subagent-voice` entry built here is exactly what closes that.

**Bias Audit** (CLAUDE.md, required — touches model selection + the Anthropic path): *parity* — the change **removes** asymmetry in two directions: it stops dropping Anthropic for an availability reason, and it removes the cage Claude had invented for the Anthropic path (a restriction never imposed on subagents doing other work — textbook vendor-asymmetry bias, this time aimed inward); *capability cost* — this **adds** capability (subagent voices) at no cost to the opencode path; *provenance* — the corrective ("don't cage it, just spawn it") came from the **maintainer**, not Claude, which is precisely why it's recorded here — Claude's unaided reflex was the cage; *harness difference named* — the one true difference is **no non-Claude capture point**, which is why a subagent turn is logged as a claim (`captured:false`), not the vendor; the "zero-tool profile" is not a valid harness difference because it is **not buildable** (platform facts above), so it cannot justify friction. Recorded against AGENTS.md's PARITY rule as another caught vendor-asymmetry instance — notable as the first pointed at the Anthropic path itself.

### Delegation: guard removed, diff recorded — DONE (2026-07-15)

**The clean-worktree guard is gone.** It was Claude's own invention (`fc17389`), not a review finding or a user decision, and the maintainer called it correctly: it was unjustified asymmetry. Coordinating two Anthropic subagents over these same files involves no snapshot, no clean-tree demand, no quarantine — and a delegated model is the same class of actor: an LLM editing files a human then reviews. Requiring a commit first blocked the actual use case (working *with* another model on live, uncommitted files) for a reason a subagent doesn't equally trigger. The docs calling the diff review "the trust boundary" was the same error in prose.
- **Superseded conclusion:** `--auto` was considered as a harness difference, then rejected after testing. For the hardened agents it is a no-op because every tool resolves to `allow` or `deny`; it only matters on fallback agents with `ask` rules. Do not use it as cover for extra friction on non-Claude paths.
- **What replaced the guard is strictly better and never blocks:** `ask.sh` snapshots the worktree as a git tree (throwaway index, so the caller's index and worktree are untouched). The recorded diff is then the model's changes *only* even on a dirty tree, and clobbered work is **recoverable** (`git checkout <tree> -- <path>`) — which the refusal never offered. `git add -A` honours `.gitignore`, so `.env`/`collab/logs` stay out of the patch.

**Three bugs found while building it, all silent:**
- **`git diff <sha>` is blind to files the model CREATED**, so a delegation that adds a file produced an **empty diff** — and the review was the whole safeguard. `delegate.md` did say to also run `git status`, but `ask.sh`'s own hint pointed at the blind command: compliance, not construction. The recorded patch is built from tree snapshots and includes created files.
- **The evidence layer logged NOTHING for any call without an explicit model.** `log.sh` parsed args with `${2:?}`, which aborts on an *empty* value, and `ask.sh` passes `--model ''` when nothing is pinned. `ask.sh` swallows the logger's stderr, so it failed silently — for every user without a configured default model. Every existing test passed `-m`; that's exactly why none caught it. **Fourth instance this session of "the guarantee held only where it was tested".**
- **`/collab:witness` could not audit a delegation at all** — the watcher reads only `collab/logs/**`, and the diff wasn't there. It could check whether Claude relayed the model's *claims* faithfully while the thing worth auditing (does the diff match the report?) stayed invisible — on the command W3 calls the highest-value target. Fills W0.4's promise of a home for "`/collab:delegate` diff snapshots".

### Command namespacing — DONE (2026-07-15)

All nine slash commands moved to `.claude/commands/collab/`, which Claude Code exposes as **`/collab:<name>`** (`/collab:consult`, `/collab:review`, …). **Why:** ClaudeCollab installs into other people's repos, where our generic names collide with the user's own commands or a bundled skill — and the collision is *silent*, because the installer's merge-not-clobber rule keeps the user's file, so the command still exists and simply isn't ours. Found live: our `/review` was colliding with a bundled `review` skill in an actual session.

**Method note worth keeping — the docs were wrong and cost us a full detour.** The official docs state "the subdirectory appears in the command description but doesn't affect the command name itself", i.e. no namespacing. On that basis every command was renamed to `collab-<verb>` (a real, uglier fix). Moving those files into `collab/` for grouping then revealed `/collab:collab-consult` in the live skill list — the subdirectory *does* namespace, the doc claim is false for this surface, and the filename prefix was redundant. Dropped it; kept the directory. **Lesson: for platform behaviour, the running system outranks the documentation — check the live skill list first.** The user had chosen "prefix everything" based on the doc claim, so the decision had to be re-put to them once the evidence changed; a silent correction would have left them thinking they'd chosen `/collab:consult` when they'd been told it was impossible.

**The opencode agent defs were left flat on purpose.** opencode namespaces subdirectory agents too (`collab/read`, invokable — both verified live), but: the `collab-<role>` names are already distinctive, the gain is cosmetic, and every `verify-collab-*.sh` parses `opencode agent list` with a terminator regex that a slash breaks — each would silently verify the *wrong* agent and still print PASS. Not worth it for four files. Doctor's agent check was fixed instead (below).

Residual risk, accepted: the namespacing is **observed, not documented**, so it could change. If it ever does, the fallback is renaming the files to `collab-<verb>` — the git history shows exactly how.

Shipped alongside two detection fixes, because a namespace makes a collision *unlikely*, not impossible:
- **`doctor.sh` checks each command is present AND ours**, cross-referencing the install manifest. Previously doctor printed "OK — all required checks passed" while a command was shadowed. It reports the count that is actually ours, because "all 9 present" beside a warning that one isn't ours is the same self-contradicting clean bill this project keeps killing.
- **`install.sh` summarises skips by name at the end**, not just inline where the warning scrolls away, and says what it means: your file was kept, ours is absent, that command is not what our docs describe.
- **`doctor.sh`'s AGENT check had the same bug and was fixed too** — it reported "present" for a def ClaudeCollab didn't install. It now cross-checks the manifest. The saving grace, worth knowing: a shadowing def that is actually *weaker* already failed `verify-collab-*.sh`'s resolved-config proof (confirmed by planting a `"*": allow` def — verify FAILS, doctor exits 1), so the guarantee held; the new check just names the cause instead of leaving it looking like our own def regressed.
- **`check-frontmatter.sh` now recurses and fails on an empty set** — a flat glob would have matched zero files after the move and passed vacuously.

#### Phase W0 — the evidence layer — **DONE (2026-07-15)**; zero model calls
No watcher until the data source exists. A watcher reading Claude's account is theatre.

**Shipped as `collab/log.sh`** (+ `ask.sh` hooks, `doctor.sh` wiring, 13 cases in `run-tests.sh`). Every W0.1-W0.8 item below is built; the design survived contact with the implementation, with two corrections found by testing rather than review:
- **`turn` had to move inside the lock.** Computed outside it, three concurrent `/collab:panel` calls all read "0 so far" and each logged itself as **turn 1**. W0.3 said `call_id` (not `run_id`+`turn`) must be the pairing key and gave parallel panel calls as the reason — that reasoning was right, and it applies to *computing* turn too, not just to pairing on it. Caught by the concurrency test, which is why that test asserts distinct turns rather than just intact JSON.
- **The hash chain cannot cover the last line.** `prev_hash` links each entry to its predecessor, so the most recent entry has no successor holding its hash — and a truncated/half-flushed final write is the *most likely* accidental corruption here, i.e. the chain's blind spot was exactly its main use case. Fixed with a per-entry `response_hash` self-check over the payload, which `verify` checks alongside the chain. Both fields were already in W0.1; only the reason to keep `response_hash` changed.

**Deliberate defaults** (each is a judgement the maintainer can flip in `collab/collab.conf.local`): logging **on** (a developer who must switch on the audit trail *before* the exchange they later want audited doesn't have one); `COLLAB_LOG_PROMPTS=full` (the brief Claude wrote is prime audit material — a `/collab:workshop` brief that editorialised is how one real bug was found — at the cost of the log holding whatever repo context was pasted in); retention **14 days**. `off` means no text *and* no digest — a knob called "off" that still fingerprints every prompt would be a lie.

**Still true, and unchanged by building it:** this is not tamper-proofing, makes no chain-of-custody claim, and anything that can write the log can rewrite the hashes.

**Dogfooded `/collab:review` on it the same day** (`openai/gpt-5.5`, read-only `collab-read`). 5 findings, **4 confirmed and fixed**, 1 refuted — plus 2 more that Claude's own verification pass turned up. The confirmed ones cluster around a single theme worth naming: *the evidence layer kept breaking its own guarantees.*
- **A failed `tee` failed the model call.** Under `pipefail` the pipeline reports the rightmost non-zero status, so an unwritable `TMPDIR` made a perfectly good call exit non-zero — logging killing the very call it exists to record. Now reads `PIPESTATUS[0]`. *(Claude's own comment on that line asserted the opposite, and was simply wrong.)*
- **`verify` only checked one direction.** A `completed` with no `started` (the call succeeded, the prompt entry was lost) printed *"every started has a completed"* — the exact reassuring-but-false verdict the contract exists to prevent. Integrity is a *pairing* property; both directions now checked.
- **"Verbatim, untruncated" wasn't.** `resp="$(cat "$f")"` strips trailing newlines and then hashed the truncated value, so `verify` would confirm a copy it had itself lost bytes from. Now byte-exact via `jq --rawfile`; `verify` hashes the same bytes via `jq -j`.
- **A single quote in `$COLLAB_LOG_DIR`** made the lock's `trap` a syntax error, stranding the lock.
- **Refuted:** "macOS `find` lacks `-maxdepth`/`-mmin`" — both are documented BSD primaries. Noted rather than acted on: macOS validation is deferred until the repo goes public, so this rests on docs, not a run. But its *suggested* fix was adopted on independent grounds — the 10s lock-timeout path used to **append unlocked**, which can tear a >PIPE_BUF line and corrupt entries that were already safely recorded. It now drops the entry instead: a gap is bounded and `verify` reports it; a torn line poisons the record around it.
- **Found by Claude's own pass, not the reviewer:** an unguarded `mktemp` under `set -e` aborted `ask.sh` outright on a full/unwritable `TMPDIR` — the same "logging must never fail the call" invariant, violated a second way, and caught only because a *test written for that invariant* failed. Also `sed -i` in the new tests is GNU-only and would break the suite on macOS (the reviewer never saw the test file).

**The lesson worth carrying:** every confirmed finding was the evidence layer failing to hold a promise *it makes in its own comments* — never fail the call, record verbatim, catch the gap. Writing the invariant down is not implementing it. The tests that pin each one are now the difference.
- **W0.1** `ask.sh` appends JSONL to a git-ignored `collab/logs/`. Fields: `timestamp`, `run_id`, `call_id`, `command`, `model`, `agent`, `session_id`, `turn`, `prompt` (see W0.6), `raw_response` (full, untruncated), `exit_code`, `response_hash`, `prev_hash`.
- **W0.2 lifecycle, corrected after capture-initialization testing:** the original design used two call entries (`started`/`completed`), but current mechanics write three: `expected-call` before capture setup, then `started`, then `completed`. Integrity requires exactly one of each per `call_id`; three calls produce nine lifecycle entries. This closes the setup-failure gap the original pair could not represent. *(Original pair: nemotron; correction: implementation tests.)*
- **W0.3 `call_id` is what pairs pre/post — not `run_id`+`turn`.** Retries and **parallel panel calls** break turn-based pairing. *(gpt; empirically real — the `/collab:workshop` dogfood itself ran 3 calls concurrently.)*
- **W0.4 `run_id` groups a whole workflow**, plus a per-run report path/index so watcher reports, metadata, integrity status and `/collab:delegate` diff snapshots have a home. *(gpt — Claude's first synthesis flattened gpt's run-*directory* to a bare field; gpt caught the weakening in round 2.)*
- **W0.5 `claude-final` entry — Claude's final user-facing answer is logged.** Without it the watcher can audit dispositions but **not the summary the developer actually saw**, which is the thing most worth auditing. *(gpt; the biggest hole in Claude's synthesis.)*
- **W0.6 Prompt privacy + retention:** `COLLAB_LOG_PROMPTS=hash|full|off` (full = best audit fidelity, but logs then hold private repo context). Session-scoped by default; define retention/cleanup — an unbounded log directory is an indefinite sensitive-data surface. *(gpt + deepseek)*
- **W0.7** `ask.sh` prints `collab: log=<path>` to stderr. *(deepseek)*
- **W0.8 `claude-disposition` entries are logged BUT are CLAIMS TO AUDIT, NOT FACTS.** Claude writing its own structured disposition record is still self-report in new clothes — the watcher must compare each claimed disposition against the raw model responses and the `claude-final` entry. *(nemotron proposed them as verifiable data; gpt and deepseek independently refuted that framing. deepseek had already bounded it in round 1: mechanical capture would need Claude Code to expose structured decision state, which it does not.)*
- **Hashes now, tamper-proofing no.** `response_hash`/`prev_hash` go in from day one — **not** as a defence against an adversarial Claude (trusted-repo model says we don't), but because retrofitting a log format later is painful and old logs would lack it, and it catches accidental corruption cheaply. No `chattr +a`, no chain-of-custody, no tamper-proof claims. *(Claude's synthesis wanted to defer all hashing; gpt and nemotron both refuted. gpt's argument is about **format evolution**, which is independent of the threat model — so it does not conflict with deepseek's "don't overbuild for a threat we don't face". ### Rewrite: TypeScript MCP server — DECIDED 2026-07-22 (maintainer); SPIKE-GATED, NOT STARTED

**The decision.** Pause new feature work on the bash layer and rewrite the plumbing in **TypeScript** as a **local MCP server fronting `opencode serve`** — gated on a spike that must pass before any rewrite work begins. Bug fixes to the bash layer continue; features are frozen. No GitHub issue yet, deliberately (maintainer, 2026-07-22) — this section is the record until the spike is scheduled.

**Provenance.** Maintainer decision, 2026-07-22, on growth grounds — not a rescue: the current setup demonstrably works (acceptance matrix, live e2e proofs, 165 green wrapper tests). The driver is the roadmap: the per-call approval bridge (above), warm-session multi-turn continuity, and W3 push-mode watching are all long-lived orchestrators over SSE/HTTP, which bash is the wrong tool for. Two independent outside evaluations (2026-07-21 `EVALUATION.md`; a second full review with empirical verification, 2026-07-22) both named bash the project's main source of *accidental* complexity — the bash-3.2 empty-array aborts, BSD `mktemp`, `$(...)` newline stripping, the triplicated `conf_get`, the global install's byte-literal command templating are bash's problems, not this tool's. The 2026-07-22 review's confirmed defects (doctor's partition-blind log check, the verify scripts proving pattern-presence rather than path-resolution, the installer-test fixture inheriting `commit.gpgsign`) cluster in exactly the cross-file seams that shell's lack of shared modules invites. Language choice: TypeScript over the maintainer's top-comfort Swift, on ecosystem fit (opencode ships via npm and has a TS SDK, the MCP SDK is TS-first, no compile/distribute story) — maintainer's explicit call, 2026-07-22, with the Swift trade-off stated rather than skipped.

**Why the MCP-server shape, not a straight port of the wrapper.**
- **The 6-variant `Bash()` grant matrix disappears.** MCP tools get one-time approval; nine command files stop carrying near-identical env-prefixed grant permutations that every new knob multiplies.
- **Capture by construction.** The server records the byte-exact prompt/response from the API envelope as it passes through — mechanically, not via `tee` hooks bolted around a subprocess. Strictly stronger than today's evidence hooks.
- **Global by nature.** A server registered once in the user's config works in every project; most of the global-install templating machinery (and its ownership-manifest complexity) stops needing to exist.
- **Structured contract.** The opencode server API replaces flag assembly, stdout scraping, the `</dev/null` stdin hack, and `--emit-session` string parsing.
- **The no-keys principle survives — this is not the PAL/Zen shape.** Those were rejected for *needing API keys*; a local server fronting opencode inherits opencode's OAuth. The rejection was about keys, never about MCP itself.

**What is preserved (this is a plumbing rewrite, not a design rewrite).** The four hardened agent defs and their permission maps (opencode-side, untouched); the command workflows and their contracts (commands survive as thin prompts driving MCP tools); the model policy design; the evidence-layer semantics (three-entry lifecycle, both-direction cardinality, byte-exactness); the security doctrine and PARITY rule; the test suites as the behavioral contract to port against.

**Costs, stated.** A persistent process (explicitly in scope per Guiding priority); distribution shifts from `curl | bash` drop-in to npm, trading per-file shell auditability for a package — the same trust story opencode itself asks users to accept; and the bash layer's paid-for portability hardening (the macOS/BSD CI class) is discarded with it.

**The spike (the gate).** One minimal TS MCP server (stdio transport, `@modelcontextprotocol/sdk` + `@opencode-ai/sdk`) exposing a single `collab_consult` tool against `opencode serve`, using the existing `collab-read` def unmodified. Exit criteria, ordered by what rides on them:
1. **Permission maps hold through the server API — the KILL criterion.** `opencode run --agent collab-read` provably enforces the default-deny allowlist; the spike must prove the *server* session path resolves and enforces the same permissions. Re-run the secret-canary and write-attempt probes from `verify-collab-read.sh` through the server. If this fails, the shape is dead and the decision reopens.
2. **Capture is byte-exact** from the API envelope, to the current log's standard.
3. **The Claude Code approval UX** works as claimed: project-scoped `.mcp.json`, one-time tool approval, no per-call prompting.
4. **Session lifecycle** — server start/attach, session create, cleanup.
Nothing else in the spike: no panel, no policy, no witness.

**Parity note (Bias Audit).** No asymmetry introduced or removed: the permission maps stay opencode-side and unchanged, model-selection rules are untouched, and this entry changes no current behavior. Provenance is recorded above; the capability cost is the "Costs, stated" list.

### Permission realignment: external LLMs get subagent-parity tools, not fences — DECIDED 2026-07-22

**North star, restated by the maintainer (clarifying the original brief).** The tool exists so Claude Code can farm **real work** — questions, code review, coding — out to **other LLMs** to break Anthropic groupthink and catch logic holes, run through a coordinating-boss workflow: plan with many agents → maintainer review → managed build with code+functional review loops → maintainer review → PR. The day-one requirements were four: (1) **easy for the developer, ideally automatic for the coding agent**; (2) Claude uses other LLMs **like its own subagents**; (3) **no bias against other LLMs** — the maintainer's own testing showed Claude agents dismissing other-LLM feedback until an Anthropic agent echoed it, at which point the same point suddenly mattered; (4) **UI visibility** into what the external agents are doing (ask #4 — never built).

**The drift diagnosis (name it: Claude's recurring failure mode).** The calls to external LLMs kept being built **over-restrictive** — `grep`/`glob` denied, secret-file globs, egress bans, the whole witness apparatus — and every instance was the **vendor-asymmetry bias in a costume**, each hard for the maintainer to push back on because each arrived dressed as a principled security control. **Root cause:** `ask.sh` runs opencode **non-interactively** (a non-TTY pipe from Claude Code's Bash tool), so there is **no mid-run human gate and no UI visibility** — and the fences were compensating for the missing **visibility** by clamping **tools**, which is the wrong fix. The MCP rewrite closes the real gap: **MCP tool calls surface natively in the Claude Code UI and get approval** (G0 check 1). That is precisely **why the fences can now come off** — the thing they stood in for now exists.

**The decision — no floor below role scoping.** "Like a subagent" means the tools that **role** has, not every tool:
- **Read-path agents** (`collab-read`: consult/panel/review/research/workshop) get **read + grep + glob + web** — exactly a review subagent. **KEEP no-write and no-`task`** (`task` escapes to a write-capable agent); that is what makes "read-only" *true*, the same scoping you'd give a Claude reviewer. It is **role definition, not a floor**.
- The **extra denials are REMOVED**: the enumerated **secret-glob read-denies** and the **`grep`/`glob` denials**. The docs already concede the secret globs were never a real boundary; the "opencode `grep` leaks content past read-denies" harness difference is **circular** — it only bites if you are fencing secrets, which you would **not** do to a Claude subagent.
- **`collab-build`** (the `/collab:delegate` write path) is **already coding-subagent parity** (edit/write/patch/bash; diff review is the boundary) — **UNCHANGED**.
- **`collab-research`** aligns to the read-path loosening (read+grep+glob+web). **`collab-watch`** is tied to the **demoted** witness (see the demotion record above) — **out of scope for new work, but not deleted**.

## Risks & mitigations (carry forward)
- **False read-only safety** → *addressed for the read path by a default-deny allowlist:* `collab-read` sets `"*": deny` and re-allows reading non-secret files plus webfetch/websearch — mutation, `grep`/`glob`, sub-agents, and secret reads are denied by construction, including tools opencode adds later. Two review rounds drove this: the readable `.env` hole, then `patch` (mutation), `grep` (secret **content** — it bypasses read denies), and `glob` (paths) — the denylist kept missing tools, so it was replaced with the allowlist. `grep`/`glob` remain denied as a concrete opencode harness limitation, not a permanent parity principle. Verified by `collab/verify-collab-read.sh` (asserts the `"*"` floor + effective actions + grep/read secret-canary runtime probes); re-run on opencode upgrades. *The `--edit` path (`collab-build`) uses the same allowlist* — re-allows only edit/write/patch/bash — but allows `bash` by choice, so its non-mutation denies are defense-in-depth, not by construction; the diff review is the boundary (verified by `collab/verify-collab-build.sh`). *Done:* both verify scripts' `--static` checks run in `collab/doctor.sh` (local); CI catches drift with the opencode-free source lint `check-agent-permissions.sh` instead (CI never runs opencode). **Lesson: allowlist with a deny floor, never a denylist, for a versioned tool surface.**
- **Destructive/secret-leaking delegation** → the clean-worktree guard was removed; `ask.sh` snapshots dirty worktrees and records the model-only patch; collab-build denies tool-native secret reads/egress (defense-in-depth — bash bypasses); trusted repos only; mandatory post-run diff review. Not a security sandbox.
- **Diversity theater** → show exact model ids; warn on duplicates; default must not silently resolve to Claude.
- **Context starvation** → package objective, constraints, decisions, scope into the external prompt; ask the model what context it's missing; attach files rather than dumping.
- **Hallucinated reviews/citations** → verify findings against code and consequential claims against sources; vote count is never a truth metric.
- **Cost/latency** → one call for normal consult/collab:review/collab:research; 2–3 only for `/collab:panel`; tell users how many calls a command makes; no silent retries.
- **Provider/CLI churn** → pin container version, keep a tested compatibility range, keep the wrapper the single narrow boundary.
